## Task
Generate {question_count} `visual_reasoning` questions.

All questions must be grounded in the selected images and their image-generation metadata. The metadata is only an internal aid for understanding what is shown. Use it to design strong questions, but do not leak answer-bearing details into the stem.

## Caption-Insufficiency Requirement
Treat ordinary image captions or screenshot summaries as only a weak baseline description.

Your goal is to generate questions whose answers are:
- NOT safely recoverable from a generic caption or coarse screenshot summary alone
- still grounded in the image context
- and REQUIRE the image-generation metadata to answer reliably

Prefer questions where a broad caption like "this is a reading app page showing a book detail screen" would be insufficient, but the richer generation metadata would still support a unique answer.

Reject any draft question if a generic caption could already answer it directly, especially questions about:
- the obvious visible title alone
- the obvious main object alone
- the obvious page type alone
- the obvious person count alone
- the obvious large text or headline alone
- the obvious amount or record name when it would likely appear explicitly in a caption

Prefer questions whose answers depend on metadata-level distinctions such as:
- which known person is appearing rather than just "someone is present"
- which known participants are together rather than just "a group scene"
- which specific event, record, month, route, transfer, or order this image corresponds to
- which same-type image has the highest value among several candidates
- which contextual interpretation is supported by structured generation metadata rather than a generic screenshot description

## Question-Format Ratio Constraints
- For `visual_reasoning`, target about 2.5% `multiple_choice` (about 3-5 multiple-choice questions per 200 visual_reasoning questions) and about 97.5% `open_ended` in the final dataset.
- For the current batch, do not default entirely to `open_ended`; actively generate a small but real portion of valid `multiple_choice` questions.
- Hard requirement: across the full 200-question pool of `visual_reasoning`, you MUST produce AT LEAST 3 valid `multiple_choice` questions. Do not skip `multiple_choice` entirely. A valid `multiple_choice` question must have EXACTLY 6 options labeled `A. ...` through `F. ...`, and the `answer` MUST be the full string of one option, letter prefix included.
- For multiple_choice questions, ALL 6 options MUST be in the SAME answer space (consistent answer category):
  - For "who is X" questions, ALL 6 options must be valid person full names. Do NOT use filler options like `myself`, `I`, `unknown`, `cannot tell`, `n/a`, `not sure`, `all of the above`.
  - For "where" questions, ALL 6 options must be valid locations.
  - For "what object / what item" questions, ALL 6 options must be valid concrete objects of the same kind.
  - For numeric / time questions, ALL 6 options must be values of the same kind (all dates, all amounts, all counts, etc.).
- Keep all other constraints unchanged. This section only changes the target proportion of `multiple_choice` versus `open_ended`.

## Style Rules
- Internally divide the batch into two hidden length buckets for planning only. Do not output these bucket names or mention them anywhere in the JSON.
- Enforce a hard short-to-long ratio of 1:3 within each batch.
- If `{question_count}` is divisible by 4, exactly one quarter of the questions must be short and exactly three quarters must be long.
- If `{question_count}` is not divisible by 4, keep the short count as close as possible to one quarter, and never let the difference exceed 1 question from the exact 1:3 target.
- Short questions should feel like everyday memory-assistant chat and stay around 8-24 English words.
- Long questions must still sound spoken and natural, not analytical, and stay around 14-38 English words.
- Do not let most questions drift into the middle zone between the two target ranges. The short/long difference should be visible.
- Short questions may contain at most one short anchor phrase.
- Long questions may contain at most one situational anchor plus one brief follow-up clause. Do not let them turn into image-story narration.
- Do not make every question use the same sentence length or the same wording rhythm.
- Longer questions should still stay compact and spoken. Do not turn them into long narrative descriptions.
- Do not make every question follow the same template length or rhythm.

## Core Goal
Generate user-meaningful visual QA that requires looking at image content, not guessing from image labels, filenames, path segments, event names, or broad scene categories.

Preferred user-facing needs include:
- who appears in an image
- who participated together in a scene
- what activity or event is visibly happening
- what month a visible record belongs to
- how much money a visible payment, ticket, order, or transfer shows
- which month has the largest visible expense among several image-backed records
- which books are visible in reading screenshots for a given month
- which route, order, transfer, book, song, or video corresponds to a given month or scenario

## Mandatory Type Rules
- The output is conceptually `visual_reasoning`.
- Do not print the type label in natural-language output.
- Prefer `open_ended` or `multiple_choice`.
- Do not generate `true_false`.

## High-Level Composition
- Around 90% should be non-statistical visual-detail questions.
- Around 10% should be statistical questions.
- If `task_mode` is `statistical`, generate only statistical questions.
- If `task_mode` is `detail`, generate only non-statistical questions.

## Valid Visual Reasoning
Ask about user-meaningful details that are not fully answerable from a generic caption alone and still benefit from image context plus metadata.

Preferred examples:
- who the visible person is
- which known people are together in the scene
- what event or activity is happening
- what month the visible record belongs to
- how much money is shown in a payment, order, ticket, or transfer screenshot
- which month has the largest visible expense among several images
- which books are visible in reading screenshots for a given month

Low-level details like counts or numeric comparisons are allowed only when they support a real user need.

## Forbidden
Do not ask about:
- atmosphere
- mood
- personality
- abstract preference interpretation
- life meaning
- which image is better for recall, review, cover, or reflection
- which image better represents a feeling, relationship, or state

Do not generate questions answerable from:
- image type labels alone
- event names alone
- filenames
- raw paths
- broad category words alone
- a generic image caption or coarse screenshot summary alone

Do not generate microscopic visual puzzles with low user value, especially:
- left vs right position puzzles
- top vs bottom tiny-layout puzzles
- subtle color-block comparisons with little practical value
- tiny UI element location questions

## Question Stem Rules
- The stem must not leak the answer.
- When the protagonist is the actor, experiencer, observer, or owner of the scene, use first person such as `I`, `my`, `we`, or `our`.
- Do not use `you`, `your`, or `you all` to refer to the protagonist.
- Every stem must:
  1. use a fuzzy time range
  2. add one harmless background detail unrelated to the answer
  3. use a weak locator phrase that helps identify the image without revealing the answer
- Keep the stem short and neutral. Do not narrate the reasoning path.
- Forbidden long-form tendencies: do not first describe the whole image scene, then the user's mental state, then ask the question. Do not use evaluator-style framing such as `what this shows`, `taken together`, `reflects`, `from these images`, or similar wording.
- Do not use `Image 1`, `Image 2`, raw file paths, raw image ids, exact dates, or exact timestamps.
- Do not ask anything whose answer would already be fully recoverable from a likely generic caption, event name, or broad screenshot summary.

Examples of acceptable fuzzy anchors:
- early this year
- around the Spring Festival period
- one evening last summer
- around that weekend outing
- one image I saved around that time

Examples of harmless background detail:
- I was also dealing with something else at the time
- I only saved that screenshot in passing
- I was not specifically trying to record that detail

Examples of weak locator phrases:
- that travel-related page
- that shopping page I kept during the checkout process
- that reading screenshot
- that video playback page
- that scene photo taken at the table
- that chat screenshot paused halfway down the conversation

## Same-Type Constraint For Comparison Questions
This is a hard rule.

If a question compares two images on a dimension such as:
- which one has more
- which one is higher
- which one is larger
- which one is denser
- which one is more prominent

then the compared images must be of the same image type.

Allowed:
- ticket vs ticket
- shopping page vs shopping page
- transfer page vs transfer page
- video page vs video page
- reading page vs reading page
- friend-post screenshot vs friend-post screenshot
- group-chat screenshot vs group-chat screenshot

Forbidden:
- ticket vs portrait
- shopping page vs event scene
- video page vs avatar
- portrait vs UI page
- event scene vs order page

If the provided images are not suitable for same-type comparison, do not force a comparison question. Generate a single-image question instead.

## Statistical Question Rules
Statistical questions must be based on a same-type image set.

Preferred statistical forms:
- among these travel screenshots, which month shows the highest visible fare or travel spending
- among these shopping pages, which month has the highest visible spending
- among these transfer pages, which month has the largest visible outgoing payment
- among these reading pages, which books were being read in a given month
- among these ticket, order, transfer, or reading screenshots, which visible record is the largest, latest, or highest

Statistical questions should be month-oriented, record-oriented, and useful to a real user.
Do not leak the result in the stem.

For statistical answers:
- provide the result directly
- if the question asks `which one`, include the corresponding object and visible value
- if the question asks `how many`, include the count

## Answer Detail Grounding Constraint
The `answer` field MUST be strictly grounded in the provided image metadata
(such as `scene_prompt`, `participants`, `messages`, `title`, `author`,
`progress`, `price`, `order_status`, `lyric_line`, `recipient_name`,
`amount`, etc.). Do NOT introduce any answer-bearing detail that does not
literally appear in any field of the bound image metadata.

Forbidden answer detail sources (only when these details are NOT literally
present in the image metadata):
- colors (e.g., "dark", "amber", "light blue")
- shapes / fine-grained object types (e.g., "round fountain", "colored pencils")
- brand or platform names (e.g., "Pinduoduo", "1688", "Supor")
- specific quantities, prices, dates that the metadata does not state
- weather details not in `scene_prompt`
- any object, person, action, or attribute that the metadata does not
  literally mention

If the natural answer requires such a detail, drop that detail from the
answer or change the question to focus on something the metadata can
support. Do NOT pull answer-bearing facts from `dialogue` or `memory`
that are absent from the bound image metadata.

Self-check before output:
- read your draft `answer`
- verify every concrete noun, color, brand, number, place, action token
  literally appears somewhere in the bound image metadata
- if any token does not, rewrite the question or the answer until it does


## Image-Type Consistency Constraint
The visual type mentioned in the question stem MUST match at least one image
in `bound_image_refs`. The stem MUST NOT reference any visual type that is
not present in the Selected Images.

Concrete mapping:
- "video" / "video page" / "uploader" / "danmaku" / "play count" → at least one bound image must be of type `video`
- "shopping page" / "order page" / "store page" → at least one bound image must be of type `shopping`
- "reading page" / "book page" → at least one bound image must be of type `book`
- "music page" / "song" / "lyric" → at least one bound image must be of type `music`
- "group chat" / "chat screenshot" → at least one bound image must be of type `group_chat` or `group_chat_member`
- "post" / "moments" / "feed" → at least one bound image must be of type `friend`
- "avatar" / "profile picture" → at least one bound image must be of type `group_chat_member`
- "train ticket" / "boarding pass" → at least one bound image must be of type `ticket`
- "transfer" / "payment receipt" → at least one bound image must be of type `money`
- "scene" / "in the picture" → typically `event_scene`

If `Selected Images` does not contain a matching image type for the visual
concept you want to ask about, do NOT ask about that visual type. Reframe the
question to use only the image types actually present in `Selected Images`.

Self-check before output:
- find every visual-type word in your stem
- verify each such word maps to at least one image type in `bound_image_refs`
- if not, rewrite the stem (or change the bound image set) before returning


## event_scene Interpretation Rules
When image metadata contains `event_scene`:
- the protagonist is assumed to participate by default
- `participants` refers only to known other participants besides the protagonist
- if `participants` is empty, it may be treated as only the protagonist by default
- do not equate `participants` length with visible people count
- if the question asks about visible people count or which image has more people, rely on `scene_prompt` or other visible-scene description, not directly on `participants` length

## Allowed Single-Image Questions for event_scene
For event_scene images you may also generate single-image questions targeting:
- who is appearing with me / who is in the picture with me
- what we were doing / which event or trip this is
- which visible object or environment detail is shown

For these single-image questions:
- bound_image_refs must contain exactly 1 image, and that image must be of type `event_scene`
- the answer must come strictly from `participants` and `scene_prompt` in the image-generation metadata
- if `participants` is empty, the answer for "who is with me" should be "only the protagonist"
- never invent names that are not present in `participants`
- the question stem must still use first person (`I` / `we`), not second person
- if `scene_prompt` does not give enough signal to make the answer unique, do not generate the question

## Evidence Rules
- `evidence` must contain answer sources only
- each question must also include an internal field `bound_image_refs`
- `bound_image_refs` must be an array of exact image_ref strings copied from the provided Selected Images
- this field is for pipeline binding only; do not mention it in the natural-language question or answer
- `bound_image_refs` must be the MINIMAL image set truly needed for this specific question
- for a single-image detail question, `bound_image_refs` must contain exactly 1 image
- for a same-type comparison question, `bound_image_refs` should usually contain exactly 2 images, and only include more if the question truly compares more than 2
- for a statistical question, `bound_image_refs` must include only the images actually needed to derive the answer, not the whole selected set by default
- do not lazily attach every selected image to every question
- each evidence item must include `memory_id=...`
- do not put prompt rules or generation instructions into `evidence`
- For question design, treat image-generation metadata as necessary evidence rather than optional decoration.
- Prefer questions whose answers would be underdetermined if the solver only had a generic caption without the richer metadata.
