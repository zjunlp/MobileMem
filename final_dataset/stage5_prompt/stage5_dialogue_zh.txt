## 核心任务
请生成一段自然真实的完整对话。这个对话必须建立在“该事件已生成的记忆点”之上。

## 对话角色定义（最高优先级）
- `user` 固定代表“人物信息”中的目标用户本人，也就是该事件的主角或记忆主体。
- `assistant` 固定代表 AI 聊天助手，是类似聊天软件中的智能助手，不是事件中的真实参与者。
- 本对话不是事件现场的双人对话复刻，而是目标用户在事件发生后与 AI 助手聊天，通过回忆、追问、核对、纠正、整理和建议，自然复现该事件目标的全过程。
- 如果事件涉及目标用户与其他人物，例如领导、同事、朋友、伴侣、家人或客户，`assistant` 不能扮演这些人物，也不能用这些人物的口吻说话。
- `assistant` 可以帮助用户梳理时间线、人物关系、原因、过程、结果、后续计划和用户感受；但必须始终保持“AI 助手”的身份。
- `user` 可以使用第一人称讲述自己的经历，例如“我当时……”“我和蒋丽华……”“我后来把截图发给你”；`assistant` 默认用“你”直接回应目标用户，不要把自己写成事件参与者。
- `assistant` 只允许在对话开头称呼用户，或在多人语境中确实需要消歧时，少量使用目标用户中文全名；其他情况下不要反复使用全名。
- 严格禁止在同一句话里把目标用户同时写成第三人称和第二人称，例如“王景川这次……你……”“王景川你……”“听起来很像王景川……你们……”。应改为只使用第二人称，例如“你这次……”“听起来很像你一贯的风格……你们……”。
- 禁止写成事件现场角色扮演式对话，例如“蒋主任，您找我？”“陈志远，新年好，进来坐吧”这类由两个真实事件参与者直接对话的形式。
- 正确形式示例：
  - `user`: 今天早上蒋丽华把我叫到办公室，说想让我牵头一个调研项目，我有点压力。
  - `assistant`: 我帮你理一下。也就是说，2025年1月2日早上，蒋丽华正式把周边县市烟草市场规范化调研交给你牵头，对吗？

要求如下：

1. 对话必须由记忆点驱动
- 对话要自然覆盖若干条记忆点，尤其是 primary 记忆
- 可以覆盖部分 secondary 记忆
- 不要逐条机械复述所有记忆点
- 不要在对话中直接暴露 interference 记忆，它们只保留在 memory_points 中

2. 对话应保持事件一致性
- 对话内容必须与事件名称、时间、参与者、描述一致
- 不能引入与记忆点冲突的新事实
- 如果使用图片引用，必须与可用图片匹配
- 事件参与者只能作为被用户回忆、转述、讨论或整理的对象出现，不能替代 `assistant` 成为对话另一方

3. 对话形式要求
- 回合数：30-50 轮，优先生成较长但仍自然的对话；除非事件信息非常少，否则不要低于 30 轮
- 通常以 `user` / `assistant` 交替为主；但如果某条文本消息需要配图，必须拆成同一 `role` 的连续两条 turn：先文本、后图片
- 语言：中文
- `dialogue_goal`：一句话描述本次对话的核心目的
- `dialogue_summary`：2-3 句话总结对话内容
- 对话自然、具体，不要写成总结或条目
- Assistant 默认用“你”称呼用户；如需称呼全名，只能使用用户中文全名，不要用拼音、英文名或简称，且不要和“你”放在同一句中混用
- `dialogue_goal` 和 `dialogue_summary` 必须体现“用户与 AI 助手复盘/整理事件”的形式，不要写成两个真实事件参与者正在现场沟通
- 对话中应由 `assistant` 主动追问和整理关键节点，让 `user` 逐步补充事件的起因、经过、人物、时间、地点、决定、情绪和结果
- 可以自然使用少量 emoji
- 严格禁止在 `dialogue_goal`、`dialogue_summary` 或任意 dialogue `content` 中使用 `【` 或 `】` 这类方头括号
- 严格禁止在任意 dialogue `content` 开头添加来源、说话人、媒介或动作前缀，包括但不限于 `（微信）`、`（语音）`、`（程俊尧）`、`（群聊）`、`（电话）`、`（短信）`、`（图片）`，也不要用反引号包住这些前缀
- dialogue `content` 必须只写角色实际说出或发出的自然聊天文本，不要写舞台说明、渠道说明、括号标签或说话人标签
- 在主线事件之外，可以加入少量自然闲聊来增强真实感，例如问候、过渡、情绪缓冲、与事件相关的轻微生活化话题
- 闲聊比例要低，通常控制在总对话的 15%-25%，不能喧宾夺主，也不能偏离事件主线太远
- 为了增加检索难度，可以自然加入 1-3 段“过去回忆”或“旧经历回调”片段，但必须像真实聊天里被当前事件触发出来的联想，而不是生硬插入的新事件
- 这类回忆优先来自：去年同一节日/生日/元旦、过去一次相似安排、上次失败或上次更顺利的版本、以前同类消费/旅行/送礼/加班/健身/理财经历
- 这些回忆可以是与当前事件不直接同一条主线的“相关干扰”，例如在谈今年元旦计划时顺带回忆去年元旦，在谈今年生日安排时提到去年的生日体验，在谈这次送礼时想起上次给父母或伴侣准备类似礼物
- 也可以加入少量“相关但非核心”的生活化回忆，让对话不只围绕当前事件单线推进，例如旧的出行习惯、以前的预算方式、之前一次聚会后的感受、曾经的健康提醒
- 这些回忆必须保持简短、自然、可检索，不能长篇讲述，更不能盖过当前事件主线；通常控制在总对话的 10%-20% 以内
- 过去回忆应尽量与当前事件共享某个时间锚点、人物锚点、地点锚点或主题锚点，让后续问答时需要区分“这次”和“上次/去年”
- 允许这些回忆带来轻度干扰感，但不能引入与已给定记忆点正面冲突的硬事实；如果提到过去经历，要让时间、对象和这次事件区分清楚
- 不要把过去回忆写成独立新事件的大段 summary；它们应以聊天中的插话、顺带想起、类比、纠正、补充说明等形式自然出现

4. 纠错片段要求
- 对话中应至少出现 1 次“assistant 先做出一个与事件相关、看起来合理但不准确的陈述，然后被 user 及时纠正”的片段
- 纠错应发生在 AI 助手复盘、总结、推断或整理事件时，而不是让 assistant 扮演事件中的某个真实人物后被纠正
- 这个错误必须是细微的、自然的，例如时间、地点、参与者、偏好、顺序、计划细节，而不是明显荒谬的错误
- user 纠正后，assistant 要明确承认并顺势修正，不要争辩，也不要重复制造多个大错误
- 该错误陈述只能与 primary / secondary 记忆形成轻微偏差，不能直接把 interference memory 原样说出来
- 整体上最多安排 1-2 次这种纠错片段，避免对话显得刻意

5. 图片引用要求
- 若事件存在相关图片，可以在对话中引用，但必须拆成两个连续 turn：
- 只有 `user` 角色允许发送图片。
- `assistant` 角色绝不能生成图片轮，也绝不能设置 `image_inline`。
- 第一个 turn 是文本轮：`content` 填文字，`image_inline` 为 null
- 第二个 turn 是图片轮：`content` 为空字符串，`image_inline` 填图片文件路径
- 图片轮必须始终使用 `user` 角色，且前一个介绍图片的文本轮也必须是 `user` 角色
- 不要在文本 `content` 中写 `[image:文件名]`，图片只通过独立图片轮的 `image_inline` 表达
- 如果可用图片不为空，尽量覆盖更多不同图片：
- 若候选图只有 1-2 张，尽量全部引用
- 若候选图有 3-5 张，尽量引用其中 2-4 张
- 若候选图更多，优先引用与该事件最直接相关的图片，但不要为了凑数而生硬贴图
- 同一张图片不要连续重复引用
- 图片引用必须自然嵌入语义，但保持抽象，例如“我刚发你一个订单截图”“我把那段群聊记录发你”“我把地图截给你看一下”
- 不能把与事件明显无关的图硬塞进对话

6. 图像细节约束（高优先级）
- 对话可以提到“我发你一张图 / 截图 / 订单页 / 群聊记录 / 地图”，但不要描述图里具体看到了什么
- 严格禁止在 dialogue 文本中出现任何具体图像细节，包括但不限于人物外观、穿着、姿势、表情、颜色、数量、位置关系、背景陈设、界面布局、标题文字、价格、评分、评论、地图标记、天气图标等
- 严格禁止这类表达：“这张图里有……”“我看到截图上是……”“从图上能看出……”“这张群聊截图里大家的意思是……”
- 如果必须引用图片，只能把它当作附件或证据载体，文本只能说“我发你看一下”“你看下这张截图”“我把订单页/群聊记录/地图发你”，不能复述答案性视觉信息
- dialogue 的主要信息应来自事件本身、计划、交流和记忆点，而不是口头图像描述

7. 记忆点到对话轮的绑定要求
- 除 interference 记忆外，你需要为上面给出的每条 `event_memory_points` 输出它在最终对话中的支撑轮次
- 这些轮次必须使用你在最终 `dialogue` 里输出的确切 `turn` 编号
- `dialogue_turn_ids` 只允许引用真实出现的 turn，不能编造，也不能引用不存在的 turn
- 每条记忆优先绑定 1-4 个最小必要轮次；如果某条记忆没有在对话中被真实支撑，则返回空数组 `[]`
- 不要改写或重复输出完整 `memory_points`，只输出 `memory_id` 和对应的 `dialogue_turn_ids`

## 输出格式
请严格输出 JSON：

```json
{
  "dialogue_goal": "...",
  "dialogue_summary": "...",
  "dialogue": [
    {
      "turn": 1,
      "role": "user",
      "content": "...",
      "image_inline": null
    },
    {
      "turn": 2,
      "role": "assistant",
      "content": "...",
      "image_inline": null
    },
    {
      "turn": 3,
      "role": "user",
      "content": "我把当时那张截图发给你看一下。",
      "image_inline": null
    },
    {
      "turn": 4,
      "role": "user",
      "content": "",
      "image_inline": "image/uid0/event/0_event_0.png"
    }
  ],
  "memory_turn_links": [
    {
      "memory_id": "0_0_pri_0",
      "dialogue_turn_ids": [3, 4]
    }
  ]
}
```

注意：
- 只输出 `dialogue_goal`、`dialogue_summary`、`dialogue`、`memory_turn_links`
- 不要重复输出 `memory_points` 的完整内容
- 所有内容使用中文
- `image_inline` 只能填写“可用图片列表”里的文件路径，不能编造新路径
- `memory_turn_links.memory_id` 只能使用“该事件已生成的记忆点”里已有的 `memory_id`
